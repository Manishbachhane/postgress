-->JSON Kya Hai?->JSON data ko text format me store karta hai.

-->JSONB Kya Hai?->JSONB same JSON data ko PostgreSQL internally optimized binary format me convert karta hai.

Main Difference 
JSON

JSONB
Optimized searchable storage


Practical Difference

->JSONB

JSON — 10 Important Facts 🚀
--->JSON data ko raw text format me store karta hai.
--->Formatting preserve ho sakti hai.
--->Key order preserve ho sakta hai.
--->Duplicate keys allow karta hai.
--->{
--->  "name":"Aman",
--->  "name":"Ravi"
--->}
--->Searching/filtering possible hai.
--->But every query me parsing hoti hai.
--->Isliye performance slower ho sakti hai.
--->Proper indexing support weak hai.
--->Mostly raw API responses store karne me use hota hai.
--->Real-world PostgreSQL apps me rarely use hota hai.

--->JSONB — 10 Important Fa
-->JSONB — 10 Important Facts 🔥
-->JSONB = Binary optimized JSON storage.
-->PostgreSQL internally optimized structure banata hai.
-->Formatting preserve nahi hoti.
-->Duplicate keys automatically normalize ho jati hain.
-->Searching/filtering much faster hoti hai.
-->Powerful indexing possible hai.
-->GIN indexes JSONB ke sath common hain.
-->Nested objects/arrays efficiently handle karta hai.
-->Real-world production apps me mostly JSONB use hota hai.
-->Flexible schema + SQL power dono deta hai.

Part 3: Dangerous Zone — Kaise use NAHI kar sakte? (Strict Rules)
Bhai, is section ko dhyan se samajhna, saari errors yahi bachti hain:

Rule 1: ->> ke aage chain nahi chalti!

attributes -> 'stock_info' ->> 'quantity' ➔ Sahi hai (Pehle waale ne JSON diya, agle ne text nikala).

attributes ->> 'stock_info' -> 'quantity' ➔ ➔ WRONG (ERROR)! Kyunki ->> ne pehle hi sab text bana diya, ab text ke andar arrow nahi ghus sakta.

Rule 2: -> ko normal string se compare nahi kar sakte!

WHERE attributes -> 'category' = 'auto_parts' ➔ WRONG! Database kahega ki left side JSON string hai "auto_parts" aur right side normal string hai 'auto_parts', dono alag hain.

Rule 3: @> me incomplete structure nahi chalega!

WHERE attributes @> '{"compatible_bikes": "Pulsar 220f"}' ➔ WRONG! Kyunki original data me compatible_bikes ek Array [] hai, toh right side me bhi array format ["Pulsar 220f"] hona zaroori hai.

Rule 4: JSON operators ke andar mathematical check direct nahi hote!

WHERE attributes ->> 'price' > 200 ➔ WRONG/UNEXPECTED! Kyunki ->> text return karta hai, aur text ka comparison text ki tarah hoga (jaise '30' bada ho jata hai '200' se alphabetical order me).

===============================
MATERIALIZED VIEW — POSTGRESQL
===============================

-----------------------------------
1. WHY MATERIALIZED VIEW EXISTS
-----------------------------------

Problem:
Heavy queries become slow when executed repeatedly.

Example:
- JOIN on multiple tables
- GROUP BY
- SUM / COUNT
- Analytics queries
- Dashboard reports

Without Materialized View:
Every request recalculates query again.

Result:
- High CPU usage
- Slow API
- Database load increase

Solution:
Store precomputed query result physically.

Materialized View:
= Saved query result on disk.


-----------------------------------
2. MENTAL MODEL
-----------------------------------

VIEW
= Live camera feed
= Always fresh
= No data stored

MATERIALIZED VIEW
= Screenshot
= Stored snapshot
= Fast reads
= Can become stale


-----------------------------------
3. NORMAL VIEW vs MATERIALIZED VIEW
-----------------------------------

VIEW:
- Stores only SQL query
- Executes query every time
- Always fresh
- Slower for heavy queries

MATERIALIZED VIEW:
- Stores final result physically
- No recalculation during read
- Very fast reads
- Data may become outdated


-----------------------------------
4. INTERNAL WORKING
-----------------------------------

CREATE MATERIALIZED VIEW:

Step 1:
PostgreSQL executes query.

Step 2:
Creates physical storage on disk.

Step 3:
Stores result rows like a table.

Example:

CREATE MATERIALIZED VIEW customer_sales AS
SELECT customer_id,
SUM(amount) AS total_sales
FROM orders
GROUP BY customer_id;


-----------------------------------
5. IMPORTANT COMMANDS
-----------------------------------

Create MV:

CREATE MATERIALIZED VIEW customer_sales AS
SELECT customer_id,
SUM(amount) AS total_sales
FROM orders
GROUP BY customer_id;


Read MV:

SELECT * FROM customer_sales;


Refresh MV:

REFRESH MATERIALIZED VIEW customer_sales;


Concurrent Refresh:

REFRESH MATERIALIZED VIEW CONCURRENTLY customer_sales;


-----------------------------------
6. MOST IMPORTANT CONCEPT
-----------------------------------

Materialized View
= SPEED vs FRESHNESS tradeoff

Fast Reads:
YES

Real-time Data:
NO

Why stale?
Because new table data does NOT automatically update MV.


-----------------------------------
7. STALE DATA EXAMPLE
-----------------------------------

orders table:

customer_id | amount
1           | 100
1           | 200

MV Result:
1 | 300

Now insert:

INSERT INTO orders VALUES (3,1,1000);

Actual total:
1300

BUT MV still shows:
300

Why?
Because MV snapshot is old.

Fix:
REFRESH MATERIALIZED VIEW customer_sales;


-----------------------------------
8. INDEXES ON MATERIALIZED VIEW
-----------------------------------

YES, indexes supported.

Why?
Because data is physically stored.

Example:

CREATE INDEX idx_customer_id
ON customer_sales(customer_id);

Benefit:
- Faster lookup
- Less scanning
- Better performance


-----------------------------------
9. CONCURRENTLY REFRESH
-----------------------------------

Normal REFRESH:
Blocks reads.

Problem:
Users cannot query MV during refresh.

Solution:

REFRESH MATERIALIZED VIEW CONCURRENTLY customer_sales;

Benefit:
Users can still read during refresh.


-----------------------------------
10. IMPORTANT CONDITION
-----------------------------------

CONCURRENTLY requires UNIQUE INDEX.

Example:

CREATE UNIQUE INDEX idx_unique_customer
ON customer_sales(customer_id);

Why?
PostgreSQL needs unique row identity internally.

-----------------------------------
13. WHEN TO USE
-----------------------------------

Use MV when:
- Heavy aggregation
- Expensive joins
- Read-heavy system
- Dashboard/reporting
- Data freshness not critical

Best for:
- Analytics
- Reporting
- Statistics


-----------------------------------
14. WHEN NOT TO USE
-----------------------------------

Do NOT use when:
- Real-time data required
- Small tables
- Data changes every second
- Frequent refresh needed

Example:
Live stock market prices.


-----------------------------------
15. BEGINNER MISTAKES
-----------------------------------

Mistake 1:
Thinking MV auto-updates.
→ FALSE

Mistake 2:
Using MV for small/simple queries.

Mistake 3:
No indexes on MV.

Mistake 4:
Refreshing too frequently.

Mistake 5:
Ignoring refresh performance cost.


-----------------------------------
16. PERFORMANCE IMPACT
-----------------------------------

Advantages:
- Extremely fast reads
- Reduced query execution time
- Less CPU during reads

Disadvantages:
- Extra storage usage
- Expensive refresh
- Stale data risk


-----------------------------------
17. VIEW vs MATERIALIZED VIEW
-----------------------------------

Feature                 VIEW      MATERIALIZED VIEW

Stores Data             NO        YES
Fast Reads              NO        YES
Always Fresh            YES       NO
Needs Refresh           NO        YES
Supports Index          NO        YES
Storage Usage           LOW       HIGH


-----------------------------------
18. BEST PRACTICES
-----------------------------------

1. Use only for expensive queries.

2. Add indexes.

3. Refresh strategically.
   Example:
   - hourly
   - nightly
   - cron job

4. Use CONCURRENTLY in production.

5. Monitor refresh time.
